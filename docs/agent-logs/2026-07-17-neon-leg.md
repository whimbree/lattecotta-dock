# KDE neon leg - multi-distro CI matrix (Phase A/B)

Bringing up the KDE neon build + sceneprobe render leg, the deb-family
KDE-enthusiast reference environment (docs/tracking/multi-distro-ci-plan.md,
DECISION 3). Branch `multi-distro-ci-phase-a-neon`. B2 (the full e2e gate
wiring) is the orchestrator/B2-owned set and is deliberately NOT done here;
this leg proves build + ctest + sceneprobe render (A1/A2/A3/B1/B3).

neon is the closest sibling to the Debian leg: both are apt/deb with the
same x86_64-linux-gnu multiarch libdir. The divergence is the package
NAMES - neon builds its own KF6/Plasma packages, so several dev names
differ from BOTH Arch and Debian/Ubuntu-official.

## Base image (A3)

The stale Docker Hub `kdeneon/plasma:*` tags are all frozen on Ubuntu
22.04 (jammy) / Plasma 5.27 from 2023 (below floor); Docker Hub is no
longer updated. The LIVE neon images are on the KDE registry:
`invent-registry.kde.org/neon/docker-images/plasma:{user,testing,
unstable,developer}`. The **User Edition** (`:user`) is the stable release
channel real neon users run, and is the pinned base for this leg. The neon
image defaults to an unprivileged `USER neon` (uid 1001); the Containerfile
switches to `USER root` for apt and the build/gate driver (parity with the
other legs, all of which run as root).

## Reproduce

```
podman pull invent-registry.kde.org/neon/docker-images/plasma:user
podman build -f ci/containers/Containerfile.neon -t latte-neon:proto .

# clean build (565/565)
podman run --rm -v "$PWD":/src:ro -v latte-neon-build:/build -w /src \
    latte-neon:proto bash /src/ci/build-and-gate.sh build

# offscreen ctest (80/80, the two NixOS-tier tests excluded by the driver)
podman run --rm -v "$PWD":/src:ro -v latte-neon-build:/build -w /src \
    latte-neon:proto bash /src/ci/build-and-gate.sh test

# sceneprobe render gate (13 scenes, nested kwin + lavapipe)
podman run --rm -e BUILD=/build -v "$PWD":/src:ro -v latte-neon-build:/build \
    -w /src latte-neon:proto bash /src/scripts/sceneprobe-gate.sh
```

## ctest characterization (A2)

`ci/build-and-gate.sh test` -> **80/80 pass** (the driver's current test
stage runs `ctest -E '^(qmllintgate|schemesmodeltest)$'`, excluding the two
NixOS-tier entries by construction). Running the full raw `ctest` shows
82 registered, 2 of which fail for the documented NixOS-tier reasons and
nothing else:

- **qmllintgate** - correctly REFUSED, not a defect: the qmllint ratchet
  structurally refuses any qmllint outside /nix/store (neon's is
  /usr/lib/qt6/bin/qmllint), because its per-file warning baseline is
  calibrated to the pinned nix qmllint. NixOS-tier merge gate; the driver
  excludes it.
- **schemesmodeltest** - non-hermetic: on the neon base (which carries the
  full Plasma desktop) /usr/share/color-schemes wins over the test's temp
  fixtures even though it pins XDG_DATA_DIRS, so the real Breeze schemes
  shift the asserted rows ("Breeze Light" where the fixture expects "Aqua
  Scheme"). Known-open, tracked in B2; the driver excludes it.

Both are the SAME pair every non-nix leg excludes; the driver already
names and excludes them (no per-leg action). All other 80 pass, including
the QML script-gates (qmlcompile/qmlinteraction/qmlcontracts) because the
image PATH carries /usr/lib/qt6/bin so qmltestrunner/qmllint resolve, and
the image ENV exports LATTE_QML_MODULE_PATH so the QML-loading tests
resolve their framework modules.

## Sceneprobe render (B3) - invariant+tolerance tier

All 13 real scenes RENDER (non-blank, correct regions/colors, self-test
valid: good passes, bad/blank fail). neon is NOT bit-exact against the
nix-blessed lavapipe goldens - it ships Mesa 25.2.8 on **LLVM 20.1.2**, a
DIFFERENT LLVM than the nix pin (and than Debian/Tumbleweed's LLVM21, which
happened to land bit-exact). 8 of 13 scenes are bit-exact; 5 differ, all by
a single-LSB-class delta:

```
    bit-exact PASS (8): badgeeffect, forced_monochromatic,
                        multieffect_brightcontrast, multieffect_colorize,
                        multieffect_desaturate, multieffect_mask,
                        multieffect_passthrough, parabolic_zoom
    DIFFER (render OK, 5): applet_colorizer        167 px (0.25%)  max Δ=1
                          indicator_glow          834 px (1.27%)  max Δ=1
                          multieffect_blur       61146 px (93.30%) max Δ=2
                          multieffect_degenerate     8 px (0.01%) max Δ=1
                          shadoweditem             64 px (0.10%)  max Δ=1
```

Max pixel delta across ALL diffs is **2** (one 8-bit level in one channel,
e.g. #ff7e0678 -> #ff800678). multieffect_blur touches 93% of pixels but
still only by Δ<=2: a Gaussian blur spreads the 1-LSB rounding across the
whole gradient, so the count is high while the magnitude stays minimal.
A `CompareTolerance{delta=2, budget>=0.95}` passes all 13 (delta covers
the LSB rounding; the high budget covers the blur's spread). This is
BIT-IDENTICAL to the Fedora leg's result (same 5 scenes, same pixel counts,
same Δ=2) - neon's LLVM20 and Fedora's LLVM21 land on the same side of the
nix pin's rounding.

sceneprobe-gate.sh exiting 1 here is CORRECT: neon simply is not a
bit-exact distro, and the gate has no tolerance-tier flag yet. Setting the
actual tolerance (or blessing a pinned-tag `*.expected.neon.png`) is Phase
C (C1/C2), not this leg. This is the second concrete proof (after Fedora)
of the graduated-rigor model's premise: bit-exactness is a per-Mesa/LLVM
accident, so neon needs the tolerance tier the plan pre-assigned it.

## Dep-name resolutions (neon vs Debian/Ubuntu-official)

Resolved by `apt-get install --dry-run` in-image against plasma:user.
neon builds its own KF6/Plasma packages, so most names match Debian but
FIVE diverge (all confirmed installing cleanly):

- **kf6-ksvg-dev** - KSvg. `libkf6svg-dev` does NOT exist on neon (Debian's
  name); neon carries a mixed libkf6*-dev / kf6-*-dev naming, and KSvg is
  the kf6- form. The genuinely surprising one.
- **kf6-qqc2-desktop-style** - the org.kde.desktop QML style. NOT Debian's
  `qml6-module-org-kde-desktop`. Confirmed it drops org.kde.desktop into
  the framework qml tree (needed by the QML gates / dock render).
- **plasma5support-dev** - `kf6-plasma5support-dev` is a neon "Dummy
  transitional"; the real one has no kf6- prefix. Debian used
  `libplasma5support-dev`.
- **plasma-activities-dev** / **plasma-activities-stats-dev** - NOT
  Debian's `libplasmaactivities-dev` / `libplasmaactivitiesstats-dev`.
- **qt6-svg-dev** - used directly (Debian's `libqt6svg6-dev` is a
  transitional dummy pointing at qt6-svg-dev on neon).

Everything else matches the Debian leg: libkf6<component>-dev for the rest
of KF6, libkirigami-dev (Kirigami), qt6-5compat-dev, qt6-shadertools-dev,
qt6-tools-dev(+-tools), libplasma-dev, plasma-workspace-dev (carries
PlasmaQuick/LibTaskManager/LibNotificationManager), kwayland-dev,
libksysguard-dev, libkpipewire-dev, liblayershellqtinterface-dev,
kwin-wayland, mesa-vulkan-drivers (lavapipe), fonts-noto-core, libcap2-bin.

## Traps / env paths (all confirmed with ls in the built image)

- Base runs as `USER neon` (uid 1001) - the Containerfile sets `USER root`
  so apt and the build/gate driver work (parity with the other legs).
- lavapipe ICD is **`/usr/share/vulkan/icd.d/lvp_icd.json`** (UNSUFFIXED),
  same as Debian.
- QML framework tree: **`/usr/lib/x86_64-linux-gnu/qt6/qml`** (Ubuntu
  multiarch libdir) -> LATTE_QML_MODULE_PATH; org.kde.desktop present.
- validation layers: `/usr/share/vulkan/explicit_layer.d`
  (VkLayer_khronos_validation.json present).
- Qt6 host tools off-PATH under **`/usr/lib/qt6/bin`** (qmltestrunner,
  qmllint) - added to the image PATH, same fix as Debian.
- kwin_wayland ships WITHOUT cap_sys_nice (getcap empty), so the `setcap -r`
  is a harmless no-op - kept for parity/robustness.
- The existing vk-suppressions.txt covers neon's Mesa: the only validation
  message is the known VK_KHR_create_renderpass2 / multiview / maintenance2
  emulation warning, already suppressed.
- `DEBIAN_FRONTEND=noninteractive` + `--no-install-recommends` +
  `rm -rf /var/lib/apt/lists/*` keep the deps layer lean and cacheable.
