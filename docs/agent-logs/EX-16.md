## Design decisions (made before first commit)

- Core: `declarativeimports/core/units/windowcycler.h`, header-only
  pure functions in `Latte::WindowCycler`. DEVIATION from the spec's
  header line (`plasmoid/plugin/units/windowcycler.h`), with reason:
  the spec's own responsibility line includes the containment-side
  task cycle twin (containment Tasks.qml `activateNextPrevTask`), and
  containment QML cannot import org.kde.latte.private.tasks. The
  EX-02 precedent applies exactly (parabolicrouter.h lives in
  declarativeimports/core/units because both sides consume it); one
  shared core plus ONE wrapper singleton in org.kde.latte.core
  replaces three mirrored QML/JS bodies. Recorded in the spec section
  too.
- Two function families in the one header, both selection-over-a-
  snapshot:
  - Group-window family (SubWindows.qml): `struct GroupWindow
    { QString winId; bool isActive; bool isMinimized; }`;
    `selectNext(windows, lastActiveWinId)`, `selectPrevious(...)`,
    `selectMinimizeTarget(...)` -> `std::optional<int>` (nullopt =
    nothing to act on; -1 survives only at the QML boundary).
    lastActiveWinId is `std::optional<QString>` - the QML -1
    sentinel dies at the wrapper.
  - Flat task-cycle family (containment Tasks.qml + plasmoid
    tools.js): `struct TaskEntry { bool isLauncher; bool isStartup;
    bool isGroupParent; int childCount; }`;
    `flattenTasksForCycling(entries)` -> `QList<TaskPosition>`
    (`TaskPosition { int row; std::optional<int> childRow; }`), and
    `selectAdjacentTask(count, std::optional<int> activeIndex,
    CycleDirection)` -> `std::optional<int>`. Active matching stays
    in the shell (live QModelIndex identity against
    tasksModel.activeTask); the core owns the expansion and the
    wraparound choice.
- Wrapper: `WindowCyclerTools` (windowcyclertools.{h,cpp}) registered
  as LatteCore singleton `WindowCycler` in lattecoreplugin.cpp,
  following ParabolicRouterTools exactly: QVariant marshalling,
  qCritical-and-refuse on malformed rows (missing keys, negative
  childCount), plugins.qmltypes entry so qmllint resolves it.
- winId discipline (8e8cdf31 family): ids are opaque QStrings in the
  core. The wrapper stringifies whatever the model handed QML
  (wayland UUID strings, X11 numeric ids -> decimal strings); "-1"
  maps to nullopt (the Qt5 lastActiveWinInGroup none-sentinel; real
  ids can never be "-1": X11 XIDs are unsigned, wayland ids are
  UUIDs).

## Defects found while extracting (fix at origin, own commits)

1. tools.js `activateNextPrevTask` is DEAD-BROKEN since Qt5
   (verified at f0ad7b23: same body): it iterates
   `taskList.children`, but no `taskList` id exists anywhere in the
   plasmoid's context chain (the plasmoid's list is `icList`;
   `taskList` is plasma-desktop applet heritage, and the only
   `taskList` id in this tree lives in the containment loader's
   separate context). First wheel event with wheelEnabled on throws
   ReferenceError and the feature silently does nothing. Fix at
   origin: assemble entries from `icList.contentItem.children`
   ordered by `task.itemIndex` (children order is creation order,
   NOT model order - main.qml:303 iterates the same collection),
   duck-filtering non-delegate children, reading the TaskItem typed
   properties. Failing-first regression qmltest in the fix commit.
2. Hand-mirror drift inside SubWindows.qml itself (the classic drift
   shape the spec names): activateNextTask guards
   `winIdList !== undefined` before reading `[0]`; the
   activatePreviousTask/minimizeTask copies read
   `WinIdList[0]` unguarded and would throw on an undefined role.
   Unified onto the guarded copy's semantics in the cutover (the
   shell builds one snapshot with the guard; EX-17's
   twin-unification precedent).

## Empty-list semantics (spec-mandated deviation from Qt5)

Qt5 next/prev on an empty group child list requested
`makeModelIndex(index, 0)` on a childless group - an invalid-index
activation request. Per the spec's test plan line, the core returns
nullopt and the shell warns (console.warn) instead of firing the
invalid request; minimize already did nothing in Qt5 (nullopt is
faithful there).

## Eliminated invalid states (step-2.5 point b; for the ledger tick)

- "no last-active window" as a magic id: lastActiveWinId is
  std::optional<QString> in the core; the -1 sentinel exists only at
  the QML boundary (wrapper maps "-1"/invalid -> nullopt).
- "no target" as index -1: all three select functions and
  selectAdjacentTask return std::optional<int>; -1 reappears only in
  the wrapper's QML-facing return.
- "child row on a plain task": TaskPosition.childRow is
  std::optional<int>; a non-group position cannot carry a child row.
- int winIds (the 8e8cdf31 family): GroupWindow.winId is QString;
  the wrapper stringifies at the boundary, so an int id cannot reach
  the comparison logic.
- direction as a bare bool: enum class CycleDirection {Next,
  Previous} (the QVariant boundary keeps bool `next` for QML, mapped
  once in the wrapper).
- negative childCount / out-of-range activeIndex: Q_ASSERT
  preconditions in the core (QT_FORCE_ASSERTS in tests), qCritical
  refusal at the wrapper boundary where QML input arrives.
