## Design

Core header (new): `plasmoid/plugin/units/launcherlistops.h`,
namespace `Latte::LauncherLists`. Pure value algebra, no QObject:

1. Stored-record parsing. `struct StoredRecord { QString url;
   QStringList activities; }` (empty activities == global record).
   `parseStoredRecords(const QStringList&) -> struct ParsedStoredList
   { QVector<StoredRecord> records; QStringList malformed; }`.
   Record grammar (libtaskmanager launcherList serialization):
   global -> plain url, activity-tagged -> "[id1,id2]\nurl".
   Malformed (has "[" but no "\n", or empty url tail) goes to
   `malformed`, never to `records` - the spec's "malformed tails as
   loud errors" (deviation from Qt5, which pushed a garbage record
   when the tail was nonempty and skipped silently when empty;
   recorded here and in the header comment).
   `launchersForActivity(records, currentActivity) -> QStringList`:
   Qt5 checked record.indexOf(currentActivity) over the WHOLE raw
   record (a uuid appearing inside the url would false-match); the
   typed parse makes that class unrepresentable - membership is over
   the parsed activity list. Recorded deviation, blessed by the
   extract-vs-pin paragraph (typed containers end the drift class).
2. Separator names. `separatorName(int no)`,
   `isSeparatorName(url)` - preserving the Qt5 quirk
   `indexOf(".desktop") !== 1` (position-1 check, NOT a contains
   check; pinned by test with a comment), and
   `freeSeparatorName(const QStringList &existingLauncherUrls) ->
   std::optional<QString>` scanning candidates 1..19 (Qt5's no<20
   cap). nullopt == exhausted (Qt5 returned "" silently; wrapper
   warns loudly and returns "" to keep the QML contract).
   Shell-side change: separatorExists() used
   tasksModel.launcherPosition(name)>=0; the pure form takes the
   parsed urls of tasksModel.launcherList. Equivalent for separator
   names (verbatim file:/// urls, no iconData/desktop-id ambiguity
   in launcherPosition's matching); recorded as the one live-query
   swap the live recipe must exercise.
3. Move planning (the spec's REDESIGN). `struct Move { int from; int
   to; }`; `movePlan(current, goal) -> std::optional<QVector<Move>>`.
   nullopt == not plannable (the lists are not permutations of each
   other - a NORMAL settling state during model churn; the caller
   restarts its timer, exactly Qt5's launcherValidPos===-1 restart).
   Planner: left-to-right pull - for each goal slot i, if
   current[i] != goal[i], one move pulls goal[i] from its current
   position j (j>i always) to i; slot i is then final. Properties:
   applying the full plan yields goal; plan length 1 for adjacent
   swaps (spec floor); applying only plan[0] and replanning strictly
   shrinks the plan - the termination guarantee the Qt5
   direction-heuristic loop lacked (its -1-splice could ping-pong).
   NOT globally minimal (LIS-based would be) - deliberate: n<=20ish,
   one move per 400ms tick, obviousness wins; comment says exactly
   that.
   Duplicate urls in either list are refused as not-plannable too,
   but the WRAPPER additionally qWarnings for duplicates (a model
   anomaly, distinct from normal settling).
4. Registries. One tested container for the four parallel string
   arrays: `class LauncherSet { enum class Equality { Exact,
   SubstringEitherWay }; bool add(url); bool remove(url); bool
   contains(url) const; int count() const; QStringList items()
   const; void clear(); }`. add is add-if-absent and returns whether
   it inserted (the QML counts increment only on real inserts);
   remove returns whether it removed (waitingLauncherRemoved signal
   fires only on real removals). SubstringEitherWay preserves the
   equals() quirk of waiting/toBeAdded: either url being a substring
   of the other, both nonempty - WHY: the same launcher travels as
   bare url and as url?iconData=... (addDroppedLauncher strips it,
   launcherUrlWithIcon carries it), and RealRemovalAnimation probes
   with both spellings; substring matching makes the two spellings
   one launcher. Test documents this with a real iconData-shaped
   case. Exact mode (immediate, toBeRemoved) is plain equality.
   Plus two keyed value registries:
   `class MoveIntentSet` (launcher -> pos; addIfAbsent(url,pos) ->
   bool, positionOf(url) -> std::optional<int>, remove(url),
   clear()) - kills the -1 sentinel of posOfLauncherToBeMoved and
   the dead "safety checker" mid-loop bailouts (list was never
   mutated inside those loops);
   `class FrozenZoomSet` (id -> zoom; setZoom upsert, zoomOf(id) ->
   std::optional<qreal>, remove(id), clear()) - Qt5's
   removeFrozenTask removed the LAST match, indistinguishable from
   first-match because setFrozenTask upserts (no duplicates
   representable); noted in a comment.

QML-facing wrappers (new file `plasmoid/plugin/launcherlistops
bridge`, actual names below; registered in lattetasksplugin.cpp per
the EX-17 precedent - lattetasksplugin.cpp is Latte-authored, not in
the vendored set):

- `LauncherListOps` stateless singleton:
  `launchersForActivity(QStringList records, QString activity)`,
  `freeSeparatorName(QStringList records)` (parses, allocates, warns
  on malformed + exhaustion), `movePlan(QStringList current,
  QStringList goal) -> QVariantList` of {from,to} maps (empty list ==
  not plannable; the QML caller distinguishes by having already
  checked launchersAreInSync).
- `TasksExtendedRegistries` creatable QObject holding the six
  registries as members; Q_INVOKABLEs mirror the QML function
  surface, returning bool for add/remove so the QML shell keeps its
  count/signal state machine visible. clearAll() for the GC timer;
  per-registry items() for the GC debug prints.

  registry entries, per the "count semantics preserved per registry"
  spec line.

  restart (settling); indexOfLayoutLauncher===-1 -> stop (launcher
  left the model), both exactly the Qt5 outcomes.

## Eliminated invalid states (for the ledger tick at merge)

- Activity-tag false match: an activity id matching inside the URL
  text (Qt5 substring-scanned the raw record) - unrepresentable
  after the typed parse.
- Garbage launcher from a malformed stored record (tag without
  newline) - refused loudly instead of shown.
- posOfLauncherToBeMoved's -1 sentinel and dead mid-loop "safety
  checker" returns - std::optional<int>.
- The validator's non-terminating reconciliation: the direction
  heuristic could mis-splice (goal.indexOf on an element not yet
  settled) and ping-pong; the pull planner strictly shrinks per
  applied move (tested).
- Registry equality drift: substring-vs-exact mode is now a named
  enum per registry instead of two lookalike helper functions.
- FrozenZoomSet duplicates (upsert-only API).
