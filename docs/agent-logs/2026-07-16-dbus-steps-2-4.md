## Design notes per candidate

All serializers follow the step-1 split: value records + inline pure
serialization in app/dbusreports.h (unit-tested without a corona),
live reads only in the collect* functions in dbusreports.cpp, Corona
methods stay one-liners. JSON schemas below are the contract the unit
tests pin.

### viewAppletsData(u) -> s (step 2)

Walks view->extendedInterface() (the viewAppletsOrder precedent), one
record per applet id in appletsOrder():

    {"id": 5, "plugin": "org.kde.latte.plasmoid", "index": 0,
     "geometry": [x, y, w, h], "isExpanded": false,
     "inScheduledDestruction": false, "lockedZoom": false,
     "colorizingBlocked": false}

- Every field has an existing home: AppletInterfaceData (id, plugin),
  indexOfApplet(id), appletIsExpanded(id),
  Plasma::Applet::destroyed() (the destroyedChanged signal
  ContainmentInterface already forwards as
  appletInScheduledDestructionChanged), m_appletsInLockedZoom and
  m_appletsDisabledColoring (private lists synced from the QML
  layoutManager; plain getters added, no parallel caches).
- geometry is the AppletQuickItem mapped to scene (= view window)
  coordinates: plasmoid->mapRectToScene(0,0,w,h) - "geometry within
  the view" per the doc.
- appletsOrder() includes justify splitter pseudo-ids that have no
  applet data; they are skipped with a comment (they are layout
  artifacts, not applets - the doc's schema is "per applet").

### activateTaskAt(u, i) (step 2)

GlobalShortcuts::activateEntryForView(view, index, Qt::META) IS the
Meta+number path: plasma-tasks fallback, the wait-for-shown deferral
when the dock is hidden, and the same
ContainmentInterface::activateEntry -> QML shortcuts host
activateEntryAtIndex invoke a badge click drives. Made public (the
add-the-getter rule applied to an action) and Corona delegates to it.
Index semantics are therefore exactly Meta+<index> (1-based visual
entry index).

### trackerData(u) -> s (step 3)

One JSON object off WindowsTracker + CurrentScreenTracker +
LastActiveWindow, all existing getters:

    {"containmentId": 1, "enabled": true,
     "activeWindowTouching": false, "activeWindowTouchingEdge": false,
     "activeWindowMaximized": false, "existsWindowActive": true,
     "existsWindowTouching": false, "existsWindowTouchingEdge": false,
     "existsWindowMaximized": false,
     "lastActiveWindowPresent": true,
     "lastActiveWindowAppName": "firefox"}

- The doc lists the four core facts; the touchingEdge pair and
  existsWindowActive are included too because they are the actual
  inputs the dodge/colorizer deciders consume (ColorizerDecider binds
  all of them) - the visibility-mode e2e this surface exists for
  asserts on exactly these. JSON design decision 1: adding fields is
  safe.
- LastActiveWindow has no appId getter; its application-identity
  field is appName (not window content - the title stays out, per the
  doc). The JSON field is named appName to say what it actually is.

### setViewVisibilityMode(u, s) (step 3)

The settings combo does latteView.visibility.mode = <mode>
(BehaviorConfig.qml:380), i.e. VisibilityManager::setMode. The D-Bus
boundary parses the mode name with a new pure inverse
(visibilityModeFromName, implemented by searching
visibilityModeName's own output so the two can never drift) and
REFUSES unknown names and "none" loudly: setMode Q_ASSERTs against
Types::None, and a D-Bus string is outside input, so the boundary
refusal is the router-wrapper pattern, not a bandaid.

### viewTasksData(u) -> s (step 4)

Per task row of each latte tasks plasmoid in the view:

    {"appletId": 5, "index": 0, "appId": "firefox",
     "launcherUrl": "applications:firefox.desktop",
     "isLauncher": false, "isGrouped": false, "childCount": 0,
     "isActive": true, "isMinimized": false,
     "demandsAttention": false, "badge": 0,
     "geometry": [x, y, w, h]}

- The model is the plasmoid's own TaskManager.TasksModel; a
  readonly property alias on the plasmoid root exposes it (the
  add-the-getter rule; ids outrank properties in QML scope so
  internal references are untouched).
- Roles are resolved BY NAME from roleNames() at collect time
  (AppId, LauncherUrlWithoutIcon, IsLauncher, IsGroupParent,
  ChildCount, IsActive, IsMinimized, IsDemandingAttention, Geometry) -
  no libtaskmanager header dependency; a missing role name qWarnings
  (drift detector), never silently defaults.
- NO window titles: appId + launcherUrl identify the task (the doc
  records why). LauncherUrlWithoutIcon, not LauncherUrl, so no icon
  payload rides along.
- geometry is the rect the delegate published via
  requestPublishDelegateGeometry (TaskItem.qml:729 publishes global
  coordinates) - the WM-facing minimize-animation rect.
- badge comes from the plasmoid root's typed badgeValueFor(launcherUrl)
  (the same BadgeMath path the canvas draws from), invoked by
  metaobject exactly like the updateBadge D-Bus path; 0 means no badge
  (the plasmoid's own convention).
- appletId is added per record (not in the doc's list) because a view
  can host more than one tasks plasmoid; records from each carry
  their applet id.
- A view with no latte tasks applet returns "[]" WITHOUT a warning:
  "no tasks plasmoid" is a legitimate state a consumer can
  distinguish via viewAppletsData plugins, not an error. A missing
  containment id warns (the precedent).

### colorizerData(u) -> s (step 4)

    {"containmentId": 1, "enabled": true, "themeColorsMode": "smart",
     "windowColorsMode": "none", "colorizerPresent": true,
     "mustBeShown": true, "applyingWindowColors": false,
     "backgroundIsBusy": false, "currentBackgroundBrightness": 42.5,
     "scheme": "kdeglobals"}

- The doc's prose ("mode, exempted or not, measured saturation
  bucket, scheme basename") mapped onto the state that actually
  exists: mode = the containment's themeColors/windowColors settings
  (read from the containment root item, where root.colorizerEnabled
  also lives); the decision in force = mustBeShown +
  applyingWindowColors off the colorizer Manager item
  (latteView.colorizer, bound in BindingsExternal.qml:169);
  the measured bucket = backgroundIsBusy + currentBackgroundBrightness
  (the SmartThemeColors BackgroundTracker measurement; -1000 =
  unmeasured, the Manager's own sentinel). Per-applet colorfulness
  exemption (IconColorfulness) is per APPLET, not per view - that is
  viewAppletsData's colorizingBlocked, not this surface.
- scheme is basenamed (QFileInfo().fileName()) per the doc.
- The colorizer item is a legitimately absent optional (the Manager
  binding lands after QML load): colorizerPresent reports it and the
  dependent fields keep record defaults; no warning, it is state.
- themeColors/windowColors enum names come from the REAL enums
  (containment/plugin/types.h, a header-only Q_GADGET, included
  relatively) so -Wswitch trips on new values like every other
  enum-name mapping in dbusreports.h.

### layoutsData() -> s (step 4)

    {"memoryUsage": "single",
     "layouts": [{"name": "My Layout", "isActive": true,
                  "activities": ["uuid"], "viewsCount": 2}]}

- Layouts::Manager::memoryUsage() + synchronizer currentLayouts();
  per layout: name(), isActive(), appliedActivities(), viewsCount().
- MemoryUsage::Current (-1) is a query sentinel that never comes back
  from memoryUsage(); the pure name map still handles it exhaustively
  ("current") so -Wswitch stays meaningful and the test pins it.

### Missing-id behavior (all reads)

Array-shaped reads return "[]", object-shaped reads return "{}", both
with the viewAppletsOrder-style qWarning. Reads never construct.

## Test-quality direction (mid-task, from Bree via the orchestrator)

The step-2.5 floor is a floor; the serializer/action tests get:

- data-driven rows over the FULL enum spaces (every visibility mode,
  edge, alignment, view type, theme/window color mode, memory-usage
  value) so the Q_UNREACHABLE switches have exhaustive per-row pins,
  and a full round-trip suite for the visibility-mode inverse;
- degenerate inputs asserted as loud refusals where the decision is
  pure (unknown/empty/case-mismatched mode names -> nullopt; "none"
  excluded from the settable parse). The missing-containment-id and
  non-tasks-view refusals live in Corona one-liners over a live
  synchronizer - no corona exists in the unit harness, so those two
  are pinned by the owed busctl smokes (bogus id -> "[]"/"{}" plus a
  journal warning) instead of a unit row;
- exact key-set pins per record type: each test serializes a record
  and compares the full sorted key list against
  docs/reference/dbus-observability-interface.md, so accidental schema drift
  fails a test, not a consumer. The step-1 ViewRecord gets the same
  key-set pin retroactively, and the step-1 enum-name tests are
  converted to data-driven rows in the same commit.
