## Eliminated invalid states (for the ledger tick at merge)

- WHICH edge is unrepresentable in the core: enum class Alignment
  { Edge, Centered } - no branch of the math reads the edge, so an
  alignment the math could misread cannot exist inside the unit. The
  raw Latte::Types::Alignment int stops at the bridge via a total
  mapping (Center/Justify -> Centered, everything else including
  out-of-domain ints -> Edge; bridge_mapsAlignmentTotally).
- clampOffset's result type (OffsetClamp{maxLength, offset}) cannot
  claim it touched minLength - the offset path never does.
- Non-finite state (NaN/inf from a hand-edited config; KConfig parses
  "nan" happily) cannot reach the core silently: the bridge refuses
  loudly and whole (no partial clamping of a corrupt state), the core
  Q_ASSERTs finiteness under QT_FORCE_ASSERTS. Qt5 silently pushed
  NaN back into the config keys.
- The inverted-bounds UB trap is designed out: for maxLength > 100 the
  bounds invert (from > to) and std::clamp would be UB; the core keeps
  the shipped min(max(from,x),to) composition with the reason in a
  comment and the healing sequence pinned
  (clampOffset_outOfRangeMaxLengthSelfHeals).

## Empirical probe: QML int property conversion

The offset slider bounds go through
`readonly property int screenLengthMaxFactor: (100 - maxLength) / 2`.
maxLength can be fractional (the ctrl+scroll smallStep 0.1 path), so the
double->int conversion semantics matter. Probed on the PINNED Qt
(qtdeclarative 6.11.0, qmltestrunner offscreen, 2026-07-16):
fractional-to-int property bindings TRUNCATE TOWARD ZERO
(10.5 -> 10, -10.5 -> -10, 10.6 -> 10, -10.6 -> -10;
`property int factor: (100 - 79) / 2` -> 10). Literal fractional
assignments are a compile error in 6.11, so only the binding path
exists. The core reproduces this with std::trunc, comment carries the
probe.

## Design

Core: `app/settings/lengthoffsetclamp.h`, header-only, pure
(namespace `Latte::Settings::LengthOffsetClamp`), doubles in percent
units exactly as the QML uses.

Types (step-2.5 discipline):
- `enum class Alignment { Edge, Centered }` - the math only ever
  distinguishes Center/Justify from everything else; which edge is
  unrepresentable in the core because it is unread. The raw
  Latte::Types int stays at the QML boundary (bridge maps
  Center||Justify -> Centered, all else -> Edge, total mapping).
- `struct LengthState { double maxLength, minLength, offset; }`
- `struct OffsetBounds { double from, to; }`

Functions (verb-first, one authority each):
- `pullOffsetOnScreen(LengthState, Alignment) -> LengthState` - THE
  deduplicated off-screen correction (the block duplicated between
  RulerMouseArea and the page's maxLength slider). Public: it is the
  unit's headline math and the tests pin it directly.
- `clampMaxLengthByStep(LengthState, step, Alignment)` - ruler wheel:
  max==min coupling, 30..100 clamp, min floor, then pullOffsetOnScreen.
- `clampMaxLengthToValue(LengthState, requested, Alignment)` - page
  slider: maxLength = max(requested, minLength, 1), then
  pullOffsetOnScreen. (Current maxLength is never read by this path -
  the signature says so.)
- `offsetSliderBounds(maxLength, Alignment) -> OffsetBounds` - the
  fromValue/toValue/screenLengthMaxFactor formulas, truncation included.
- `clampOffset(maxLength, requestedOffset, Alignment) ->
  { maxLength, offset }` - offset slider: bound the request via
  min(max(from, x), to) (NOT std::clamp - from>to is reachable with an
  out-of-range hand-edited config maxLength>100 and the shipped QML's
  composition order defines the result; std::clamp would be UB), then
  the maxLength-shrink correction.

Bridge: `app/settings/lengthoffsetclampbridge.{h,cpp}`,
`Latte::Settings::LengthOffsetClampBridge` (QObject). Registered as
context property `lengthClamp` in SubConfigView::init() next to
viewConfig/universalSettings - the recorded seam decision: the spec
offered universalSettings or the config view context; the config view
context wins because (a) it is registered unconditionally for BOTH
engines that need it (PrimaryConfigView hosts AppearanceConfig.qml,
CanvasConfigView hosts RulerMouseArea.qml - both are SubConfigView
subclasses sharing init()), and (b) the state being clamped is per-view
configuration, not universal app state, so universalSettings is the
wrong altitude. A dedicated small QObject rather than Q_INVOKABLEs on
SubConfigView itself because SubConfigView (QQuickView + corona deps)
cannot compile into the sanitized unit test, and the boundary refusal
must run instrumented (EX-17 wrapper precedent).

Bridge API (QVariantMap out, keys named like the config keys):
- `clampMaxLengthByStep(maxLength, minLength, offset, step, alignment)`
- `clampMaxLengthToValue(requestedMaxLength, minLength, offset, alignment)`
- `clampOffset(maxLength, requestedOffset, alignment)`
- `offsetSliderFrom(maxLength, alignment)` / `offsetSliderTo(...)`
Boundary refusal: non-finite input (NaN/inf) is refused loudly
(qCritical + input state returned unchanged). Reachable from outside:
KConfig happily parses "nan" into a double, and the Qt5 QML would have
silently propagated NaN into the config (Math.max(NaN,..) comparisons
all fail). Core Q_ASSERTs finiteness (QT_FORCE_ASSERTS keeps that live
under test); the bridge is the runtime boundary.

## Deviations from Qt5 (recorded, each with a reason)

1. The page slider's off-screen correction is computed from the
   EFFECTIVE (floored) maxLength, not the raw slider value. Qt5 wrote
   maxLength = max(value, minLength, 1) but ran the correction on raw
   `value`. The two disagree only when value < max(minLength, 1), a
   state the slider wiring makes unreachable (its pressed-branch snaps
   value up to the floor before release), so live behavior is
   identical; but the raw form breaks the on-screen guarantee and
   idempotence in that corner (counterexample in the test), and the
   spec's test plan demands both. Fix at origin, not a carried bandaid.
2. The page slider's trailing `if (maxLength < minLength)
   minLengthSlider.updateMinLength()` is DELETED as dead code:
   maxLength was just written to max(value, minLength, 1) >= minLength,
   so the condition can never hold. Dead since Qt5 (same body at
   f0ad7b23). The core's min<=max post-condition test pins the proof.
3. Same-value config writes are guarded with !== at the two QML call
   sites where Qt5 wrote unconditionally inside taken branches
   (minLength/offset on the ruler, offset/maxLength on the page).
   QQmlPropertyMap emits valueChanged on every insert including
   same-value ones; the guards strictly remove redundant emissions and
   change nothing observable (the guarded value is the one just
   computed from the same inputs).
4. NaN/inf refusal at the bridge (see above) - new behavior; Qt5
   silently corrupted the config keys in this case.

## What stays in QML (recorded scope decisions)

- minLengthSlider.updateMinLength(): pure slider wiring (writes value,
  triggers the max slider's update); contains no duplicated math. Left
  intact per the spec's source list, which names only the two
  updateMaxLength bodies and the offset cluster.
- The max slider's pressed-branch view snap
  (`value = Math.max(minLength, localMinValue)`): view state, not
  config math; the only shared expression is the trivial floor.
- Slider readiness gates (pressed, viewConfig.isReady, sliderIsReady,
  userInputIsValid): view/lifecycle state, exactly the ambient state
  the pure-core rules park outside.

## Why there is no offscreen qmltest for the shells

The tests/qml harness (qmltestrunner) cannot drive these two shells:
config-dialog QML resolves plasmoid, viewConfig, lengthClamp, dialog
and universalSettings as C++ context properties of the SubConfigView
engines, and qmltest cannot inject context properties. A harness that
faked viewConfig/lengthClamp would mock the exact logic under test
(the bridge), which docs/reference/TESTING.md bans as dishonest coverage. The
bridge instead compiles INTO the sanitized unit test (the EX-17
wrapper precedent) so the QML-facing surface runs instrumented, and
the live recipe below exercises the real wiring end to end.
