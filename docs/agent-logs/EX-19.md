finding" instruction):

## Design (recorded before code, per the execution prompt)

Core: declarativeimports/core/units/colortools.h, namespace
Latte::ColorTools, header-only pure functions:
- `colorBrightnessFromRGB(double r, double g, double b) -> double` -
  AERT brightness, 0..255 channel domain, Q_ASSERT domain preconditions.
- `colorBrightness(const QColor &) -> double` - JS semantics exactly:
  channels are double(redF())*255 (QML color.r is redF(), a float; the
  float quantization is part of the pinned behavior).
- `colorBrightness(QRgb) -> double` - integer-channel entry for the
  backgroundcache pixel loop (exact for 8-bit; agrees with the QColor
  entry to ~1e-5, the float quantization delta - documented in-header).
- `colorLuminaFromRGB(double r, double g, double b) -> double` - WCAG
  relative luminance, 0..1 domain, the 0.03928 branch per channel.
- `colorLumina(const QColor &) -> double`.
- `isLight(const QColor &, double threshold = 127.5) -> bool` -
  strictly-greater brightness comparison; matches every `> 127.5` site.
  (Spec names this exact signature.)
- QColor entries Q_ASSERT isValid() (test-time tripwire; the QML
  boundary refuses loudly in the wrapper instead).

Wrapper: the EXISTING Latte::Tools singleton (declarativeimports/core/
tools.{h,cpp}) becomes a thin shell over the core - float bodies
deleted, returns become double (the JS copies are the dedup authority
and they are double; recorded as a deliberate precision change for the
one existing consumer, SettingsOverlay). Adds Q_INVOKABLE isLight.
Invalid QColor arriving from QML (the singleton is public API for
third-party indicator packages) is refused loudly: qCritical naming
the caller's mistake, return 0.0 (the inherited silent value, now
loud). plugins.qmltypes Tools entry updated by hand (float -> double,
+ isLight), consistent with how the ParabolicMath entries were added.

App cutover: the color functions are DELETED from app/tools/
commontools.{h,cpp}; backgroundcache.cpp, theme.cpp, patternwidget.cpp
include the core header directly (single authority; float -> double at
those sites is the same recorded precision change).

QML cutovers (per-consumer commits, spec-sanctioned): each deletes the
local math and calls LatteCore.Tools; the three JS files are deleted
with their last importer. Files needing a new
`import org.kde.latte.core 0.2 as LatteCore`: AddItem, AddingArea,
ShortcutBadge. Manager.qml:55 editModeTextColorIsBright cuts to
Tools.isLight (exact same strictly-greater semantics); every other
site keeps its brightness property and comparison shape (Qt5-faithful,
no reinterpretation).

DEVIATIONS from the spec's interface sketch (recorded in the spec
section too):

## Findings so far

- The three JS copies do not diverge from each other or from Qt5
  (byte-level check above); the C++ float copies DO diverge from the
  JS in precision/channel semantics - dedup resolves to the JS double
  semantics per spec, a ~1e-5-relative change for the C++ consumers
  (theme contrast pick, background pixel averaging, pattern widget,
  SettingsOverlay) - far below every threshold compared against.
- ShortcutBadge.qml carried a real defect: the id is typo'd
  `shorcutBadge` while the Connections handler and BadgeText's
  borderColor spell it correctly - VERIFIED with an offscreen
  qmltestrunner probe replicating the shape (handler ReferenceError,
  fixedIndex stuck at -1, border color stuck at default). Fixed at the
  source (id renamed) in 8ee3b3bb, its own commit.
- LatteIndicator.qml `glow3D: glow3D` (lines 102/217, Qt5-verbatim):
  the RHS resolves to the GlowPoint's OWN glow3D, not root's - probed
  offscreen: a self-referencing binding evaluates to the property's
  existing default (true) with NO binding-loop warning, so the
  indicator's 3D-glow config value never reaches the GlowPoints
  except through showBorder's glowEnabled term. Qt5 had the identical
  resolution, so shipped behavior matches Qt5 exactly - left
  untouched per behavior fidelity (a "fix" would visibly change glow
  rendering vs Qt5). Recorded here so a future glow investigation
  starts from this fact. NOT counted by qmllint (own-property
  resolution is "qualified enough" for it).
- AddingArea.qml `appletItem.animationTime` in the opacity Behavior:
  Qt5-dead reference (sole instantiation, plasmoid main.qml, has no
  appletItem in context; Qt5-identical). Behavior duration has always
  been NumberAnimation's default. Left as counted residue named in
  section D - changing it changes observed timing.
