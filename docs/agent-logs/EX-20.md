## Design decisions (made before first commit)

- Core: `declarativeimports/core/units/badgemath.h`, header-only pure
  functions in `Latte::BadgeMath`. DEVIATION from the spec's
  `plasmoid/plugin/units/badgemath.h` placement, with reason: the arc
  geometry and the 9999+ label live in BadgeText.qml, which is
  org.kde.latte.components - a shared component the containment's
  ShortcutBadge instantiates too. Serving it from the tasks-private
  plugin would make the shared components module depend on
  org.kde.latte.private.tasks; LatteCore is the module every consumer
  already imports (the EX-19 spec makes the same placement argument
  for colortools, and declarativeimports/core/units/README.md's
  charter is exactly "logic both the containment and the plasmoid
  consume"). One core, one wrapper, all call sites reachable.
- Wrapper: `BadgeMathTools` singleton registered as
  `org.kde.latte.core/BadgeMath 0.2` (the ParabolicMathTools shape),
  files `declarativeimports/core/badgemathtools.{h,cpp}`. Stateless
  Q_INVOKABLEs; plugins.qmltypes hand-extended in the existing
  qmlplugindump format so qmllint resolves the new singleton.
- State stays in QML: root's `badgers` record list remains the
  plasmoid's per-instance state (a singleton holding it would be
  shared per-engine - wrong ownership). The core is pure over it:
  `applyBadge(records, identifier, value) -> records'`.

### Core interface

- `launcherTail(identifier)` - substring after the last '/'
  (getBadger's parse).
- `desktopEntryId(identifier)` - identifier + ".desktop".
- `struct BadgeRecord { QString id; QString value; }`.
- `findBadgeRecord(records, identifier) -> std::optional<qsizetype>`
  - getBadger's lookup: first record whose id is CONTAINED in the
  tail (substring semantics preserved, not equality).
- `parseBadgeCount(value) -> std::optional<int>` - the one value
  parse (see the unification note below).
- `applyBadge(records, identifier, value)` - update-or-append under
  desktopEntryId.
- `proportionFor(progressVisible, progress, countVisible,
  badgeIndicator) -> double` - ProgressOverlay's decision:
  progress/100, else 1 when badge/count shows, else 0.
- `arcFor(proportion) -> BadgeArc{startRadian, sweepRadian}` -
  startRadian = -pi/2, sweep = 2*pi*proportion. The canvas keeps its
  edge-bleed `filler` (presentation, canvas-specific). Q_ASSERT
  proportion >= 0 (spec: negative is a producer bug); >1 stays
  UNCLAMPED like Qt5 (canvas wraps; smartlauncheritem.cpp already
  clamps progress to 0..100 upstream of this, so >1 cannot arrive
  from the shipped pipeline).
- `ringRadiiFor(height, fullCircle) -> RingRadii{outer, inner}` -
  outer = max(0, height/2) (Qt5's partSize<0 clamp kept as a
  deliberate contract: bindings evaluate transiently during creation
  before anchors settle, a negative height there is a normal state,
  commented at the site), inner = fullCircle ? 0 : outer*0.9.
- `enum class BadgeLabelKind { None, Number, Overflow }` +
  `classifyBadgeLabel(numberValue)` (>9999 Overflow, >0 Number) +
  `formatBadgeCount(numberValue, locale)` (locale grouping, 0
  decimals). i18nc for "9,999+" is ambient state -> wrapper
  (`countLabel(int)`), per the EX-17 plan/compose split.

### Behavior calls (each recorded in the spec section too)

- VALUE-PARSE UNIFICATION (deliberate deviation, recorded): Qt5 ran
  TWO different coercions on the same stored badge value -
  main.qml's updateBadge did JS `Number(value)` (so "2abc" -> NaN ->
  0) while TaskItem's re-derive on launcherUrl change did
  `parseInt(badger.value)` (so "2abc" -> 2). The displayed badge for
  a malformed value depended on which path ran last - same input,
  two behaviors, which is a defect by this initiative's standards.
  Unified on one parse: trimmed empty -> 0 (Qt5's `value === "" ? 0`
  branch), integer/decimal strings -> truncation toward zero (both
  Qt5 paths agree on well-formed input), anything else ->
  std::nullopt, which the WRAPPER refuses loudly (qWarning naming
  identifier and value) and maps to 0 - Qt5's visible result on the
  D-Bus path. Not emulated from JS Number(): hex strings ("0x10")
  and ToInt32's 2^32 wraparound for absurd magnitudes - garbage
  domain, loud instead of faithful.
- updateBadge's METAOBJECT SIGNATURE IS A CONTRACT:
  app/view/containmentinterface.cpp:224 resolves the QML method by
  `indexOfMethod("updateBadge(QVariant,QVariant)")`. Typing the QML
  parameters would change the signature to (QString,QString) and
  silently break the whole D-Bus badge path (corona
  updateDockItemBadge -> globalshortcuts -> containmentinterface).
  Parameters stay untyped BY CONTRACT, commented at the site - same
  documented-exception class as winId vars.
- Record-store semantics preserved exactly: the record is only
  stored/updated when at least one live task matched the identifier
  (Qt5 updated it inside the task loop; zero matches = no record).
  Shell keeps the live-item loop and sets `matched`; core applies
  the record change once.
- LABEL i18nc MOVES TO C++ (wrapper): checked every Messages.sh -
  none scans declarativeimports/, so BadgeText's "9,999+" msgid has
  never been extracted into any catalog since upstream; it renders
  the English msgid today and keeps rendering it from the wrapper.
  No user-visible change; noted so a future i18n pass knows the gap
  predates this move.

  class, no sentinel strings; formatBadgeCount asserts the Number
  domain (proof slots: classifyBadgeLabel_boundaries,
  formatBadgeCount_localeGrouping, wrapper_countLabel).
