## Design decisions (made before first commit)

- Core: `plasmoid/plugin/units/scrollmath.h`, header-only pure
  functions in `Latte::ScrollMath`. One axis: the shell resolves the
  orientation (contentX vs contentY, cP.x vs cP.y, width vs height)
  and hands the core axis-scalar values, so the core has no
  vertical/horizontal branches at all - the spec sketch's
  `bool vertical` in ScrollEnv is dropped for that recorded reason
  (the QML's orientation branches are pure coordinate selection,
  structurally identical on both sides; extracting them would
  duplicate every function's math across two branches again).
  - `ScrollEnv { bool scrollingEnabled; double contentLength;
    int viewportLength; double currentPos; }` - contentLength is
    root.tasksLength (real), viewportLength the flickable's `length`
    (int property in QML).
  - `contentsExceed(env)` / `contentsExtraSpace(env)`: the overflow
    predicate and the extra-space size. These live OUTSIDE the spec's
    named line range (ScrollableList.qml:30-31 bindings, spec names
    84-187), folded in deliberately: steppedPos/focus/autoscroll all
    derive their clamp bound from extraSpace, so leaving the QML
    bindings computing it separately would leave TWO authorities for
    the overflow math - the single-copy rule forces the fold. The
    Math.floor int-world comparison and the QML int-property
    truncation (ECMAScript ToInt32, trunc toward zero) are pinned in
    the core and its tests.
  - `steppedPos(env, signedStep)`: one signed step replaces the
    increase/decrease twins; positive steps clamp at scrollLastPos
    only, negative at scrollFirstPos only - exactly Qt5's asymmetric
    min/max pair (increasePosWithStep never clamped below, decrease
    never above).
  - `wheelScrollStep(totalsLength)`: the 3.5x wheel step. Extracted
    with the rest because the same 3.5 factor is the animated
    autoscroll step multiple inside autoScrollDelta - the constant
    lives once (kScrollStepLengthFactor).
  - `focusScrollDelta(env, ItemSpan, margin) -> std::optional<double>`:
    the scroll-into-view distance; nullopt = no scroll (item visible
    or no overflow). Sign encodes direction (negative = toward the
    row start), feeding steppedPos directly.
  - `autoScrollDelta(env, ItemSpan, triggerZone, AutoScrollGuards,
    scrollAnimationRunning, totalsLength) -> std::optional<double>`:
    the trigger zones, the block conditions and the local-step
    selection, one decision. AutoScrollGuards carries
    autoScrollTasksEnabled/duringDragging/tasksCount/
    hoveredIsLastVisibleItem/parabolicZoomFactor.
  - `boundsCorrection(env) -> std::optional<double>`: the
    onContentsExtraSpaceChanged clamp; nullopt = in bounds, no write.
    Returned-as-optional so the shell only writes contentX/contentY
    when the original wrote (an unconditional clamped write would go
    through the Behavior animation even when in bounds).
- Wrapper: `Latte::Tasks::ScrollOverflowMath` QML singleton in
  lattetasksplugin (org.kde.latte.private.tasks 0.1), the
  TooltipTextComposer shape: stateless Q_INVOKABLEs, plain positional
  args for the small functions (3-6 args), a QVariantMap facts bag
  with loud missing-key refusal (composeSubText precedent) for
  autoScrollDelta's 13 facts - 13 positional number/bool args would
  make silent arg-order bugs invisible at the call site.
  Optional-returning functions return QVariant (invalid = undefined
  in QML); the none-sentinel exists only at the QML boundary, per the
  standing rule.
- Viewport unification (recorded deviation): Qt5 focusOn/autoScrollFor
  compared against scrollableList.width/height while the clamp bounds
  derive from `length`. main.qml binds the scroll-axis extent FROM
  length (`width: !root.vertical ? length : thickness`, height
  mirrored), so the two reads are the same value; the core takes one
  viewportLength and the shell passes `length` everywhere. Comment at
  the shell records the equality.
- QML keeps: fact assembly (metrics/parabolic/indexer reads,
  mapToItem, autoScrollTriggerLength = iconSize + lengthEdge),
  orientation resolution, the contentX/contentY writes (Behavior
  animations stay attached), and the public function surface
  (increasePos/decreasePos/WithStep/focusOn/autoScrollFor keep their
  signatures - TaskItem/TaskMouseArea call sites are untouched).
- Exact float boundary comparisons (currentPos !== scrollFirstPos/
  scrollLastPos) are preserved as exact != in the core with a comment:
  the pos was set by clamping to exactly those values, so exact
  equality is the Qt5-meaningful check, not a float-comparison bug.
- No defects flagged by the spec in the extracted range; the inherited
  Qt5 "TWO IN-QUESTION issues" comment on autoScrollFor moves to the
  core with the block conditions it documents.

## Eliminated invalid states (for the ledger's executed entry)

- "No scroll" is not representable as a zero delta: focusScrollDelta,
  autoScrollDelta and boundsCorrection return std::optional<double>;
  the undefined sentinel exists only at the QML boundary (the wrapper's
  invalid-QVariant conversion).
- Direction and magnitude travel as ONE signed value feeding
  steppedPos directly - "scroll toward the start by a negative
  amount" (double negation) has no encoding.
- An in-bounds position produces NO write in the extra-space clamp
  (optional again): Qt5 only wrote when out of bounds, and an
  unconditional clamped write would run through the Behavior
  animation; the no-write case is a type-level alternative, not a
  value comparison at the call site.
- A negative extra space is unrepresentable: overflow implies
  content >= viewport + 1, so the truncated difference is >= 1, and
  the non-overflow branch returns exactly 0 (asserted preconditions
  keep negative lengths out of the core; the wrapper refuses them
  loudly at the boundary with qCritical).
- The autoscroll facts cannot be silently defaulted: the wrapper
  refuses a missing key loudly (composeSubText precedent) instead of
  deciding from zero-initialized guards.

## Findings (out of scope, recorded not fixed)

- ScrollableList.qml's non-center anchor states (leftTop, leftBottom,
  rightTop, rightBottom, bottomLeft, bottomRight, topLeft, topRight)
  read `flickableContainer.lastMargin`, a property that does not exist
  on the component and is never provided by main.qml - identical at
  f0ad7b23, so Qt5 shipped the same dangling read (assigning undefined
  to an anchor margin leaves it untouched). Anchor-state code, not
  scroll math: outside EX-21's responsibility line. Left for a future
  edge-alignment unit or fix pass.
